### What's new and working today
`missing_imputation.evaluation` exposes four evaluators, all defaulting to
scoring the classical `METHODS` registry but accepting any
`{name: apply_*_imputation}` mapping via `methods=`:

| Function | Measures |
|---|---|
| `evaluate_with_sparse_validation` | Fold-wise masking CV → MAE/RMSE + full classification suite (accuracy, AUC, QWK, within-1, sens/spec/PPV/NPV) |
| `evaluate_trajectory_fidelity` | Per-patient middle-visit recovery (MAE + correlation) |
| `evaluate_temporal_smoothness` | Visit-to-visit jump penalty on imputed frames |
| `evaluate_missing_pattern_robustness` | Monotone vs intermittent missingness |

Plus a `run_all_methods` helper. All verified end-to-end on synthetic data; the
benchmark example produces a sensible leaderboard (SoftImpute/KNN beat
mean/median on QWK 0.62–0.63 vs 0.00 and within-1 0.96 vs 0.85).

## Decisions Made

- **Pivoted away from deep-learning extraction this session (key call).** The
  next-session plan called for extracting VAE / deep autoencoder / BayesianPCA /
  LSTM. However, **torch is not installed in the packaging environment**, so
  those methods cannot be executed or verified here. Shipping ~1,500 lines of
  unrunnable, verbatim-copied torch code into a public artifact is the wrong
  trade-off. I instead extracted the **evaluation suite**, which runs on the
  core deps (numpy/pandas/sklearn) and is fully testable — and which is arguably
  the more valuable public capability (it lets researchers benchmark any method
  on their own data). Deep methods remain the top item for a torch-enabled
  session (see Blockers).
- **Did NOT commit `data/` or strip notebook outputs despite the PHI clearance.**
  "No patient-identifiable data" removes the hard safety blocker, but
  de-identified clinical data is still typically kept private; whether it should
  enter the public repo is a PI decision, not a packaging default. `data/`
  remains gitignored. Notebook output stripping is likewise deferred to PI
  direction (it is destructive of research results).
- **Refactored the hard-coded notebook method lists into a `methods` parameter**
  defaulting to the `METHODS` registry. The notebook evaluators called
  `apply_lstm_imputation`, `apply_gemma_imputation`, etc., which do not exist in
  the package. Parameterising was required to make the code importable; core
  scoring logic is preserved verbatim.
- **Dropped the unused `imputed_dfs` argument** from `evaluate_trajectory_fidelity`
  (the notebook signature took it but ran methods internally), and added
  `run_all_methods` to produce the `{name: imputed_df}` mapping that
  `evaluate_temporal_smoothness` genuinely consumes.
- **Guarded the harness's heavy progress prints behind `verbose=False`** so the
  functions are library-friendly. Source: `notebook/Timeseries/imputation21`
  (the most recent/complete version) and `imputation9` for the QWK/within-1
  formulas.
